When to use chmod 640

Use 640 for: config files that a service group needs to read (like web server configs), log files readable by a monitoring group, application config files where the app runs as owner but monitoring tools are in the group.

Security note: 640 is very safe and follows the principle of least privilege well. Group gets minimum needed access (read), others get nothing.

Consider instead: If nobody else should read: use 600. If others need to read too: use 644. If the group needs write: use 660.

Common Mistake

Not using 640 enough. Many admins default to 644 when 640 would be more appropriate — keeping files private from unrelated system users while allowing the intended service group to read them.

chmod 640 vs 644

640 vs 644
640 vs 644: 640 blocks 'others' from reading, 644 allows it. Use 640 for semi-private files (config files, logs), use 644 for public-facing files (web pages, images).

Frequently Asked Questions

Should Apache/Nginx configs be 640?
Yes, 640 is ideal. The web server (running in its group) can read the config, but other system users can't see your server configuration, virtual hosts, or SSL paths.
What about log files?
640 with a monitoring group is excellent for log files. The application writes (as owner), monitoring tools read (as group), and random users can't snoop on logs that might contain sensitive data.
Is 640 the same as 'read-only for group'?
Yes. The group can only read. The owner can read and write. Everyone else is completely blocked. It's one of the most practical permissions for server administration.
