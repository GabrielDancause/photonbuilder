Permission Breakdown

When to use chmod 666

Rarely appropriate. Some legacy applications or specific development workflows use 666 for shared data files that multiple processes from different users need to write to. /dev/null is famously 666.

Security note: Any user can modify the file. On a web server, this means a compromised script could overwrite the file with malicious content. In multi-user environments, any user could corrupt shared data.

Consider instead: Use 664 for group-writable files. Use 644 for owner-writable files. If you need everyone to write, consider using a shared group with 664 + SGID instead.

Common Mistake

Using 666 for uploaded files on a web server. This allows any process to overwrite uploads. Use 644 for uploaded files, and make sure the upload directory is only writable by the web server user.

chmod 666 vs 644

666 vs 644
666 vs 644: 666 lets everyone write, 644 lets only the owner write. There's almost no good reason to use 666 instead of 644 in production. If others need to write, create a proper group and use 664.

Frequently Asked Questions

Is chmod 666 dangerous?
Yes, for most files. It allows any user to modify the file. Unlike 777, it doesn't grant execute permission, so at least nobody can run it as a script — but the write access is still a security risk.
Why is /dev/null set to 666?
/dev/null is a special device that discards all data written to it. It's 666 because every process needs to be able to write to it (to discard output). Reading it always returns empty. This is one of the few legitimate uses of 666.
What's the difference between 666 and 777?
Execute permission. 777 allows execution (running as a script/program), 666 doesn't. Both are dangerous because they allow universal write access. Neither should be used for regular files.
