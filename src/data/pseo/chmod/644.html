When to use chmod 644

Use 644 for virtually all non-executable files: web pages (.html, .css, .js), config files (that don't contain secrets), documents, images, and any file that should be readable by services like a web server but only editable by the owner.

Security note: 644 is safe for files that don't contain sensitive information. If a file has passwords, API keys, or private data, use 600 instead so only the owner can read it.

Consider instead: For files with secrets: use 600. For executable scripts: use 755. For files the group should edit: use 664. For files nobody else should read: use 600.

Common Mistake

Using 644 for files containing database passwords or API keys. These files should be 600 so only the application's owner user can read them. Many security breaches start with config files readable by all users.

chmod 644 vs 664

644 vs 664
644 vs 664: The only difference is that 664 gives the group write permission too. Use 664 when a team shares files and multiple people need to edit them. Use 644 when only the owner should edit.

Frequently Asked Questions

Is 644 the default file permission?
It depends on the system's umask, but 644 is the most common default for new files created by regular users. The typical umask of 022 produces 644 for files.
Should PHP files be 644?
Yes, PHP files should be 644 when served by Apache/Nginx with mod_php or PHP-FPM. The web server reads them — it doesn't execute them directly. Only CGI-based PHP setups need 755.
What about .env files?
.env files contain secrets and should be 600, not 644. With 644, any user on the server can read your database passwords, API keys, and other credentials.
