When to use chmod 775

Use 775 for shared project directories, team deployment folders, group-writable web directories (when developers are in the web server's group), and any directory where multiple users need to add or modify files.

Security note: 775 is moderately safe. The group write permission means any user in the file's group can modify or delete it. Make sure group membership is tightly controlled.

Consider instead: If only the owner should write: use 755. If others shouldn't access at all: use 770. For individual files (not directories): consider 664.

Common Mistake

Using 775 without setting the SGID bit on directories. Without SGID (chmod g+s), new files created in the directory may not inherit the group, causing permission issues for other team members.

chmod 775 vs 755

775 vs 755
775 vs 755: The only difference is group write permission. 755 = only the owner can modify. 775 = the owner and group members can modify. Use 775 for team collaboration, 755 for single-owner files.

Frequently Asked Questions

When is 775 better than 777?
Always. 775 gives your team full access while keeping random other users read-only. 777 gives everyone write access, which is almost never what you want.
Should I use 775 with SGID?
Yes, for shared directories. Run chmod g+s directory/ so new files inherit the group. Without SGID, new files get the creator's default group, not the directory's group.
Is 775 safe for web directories?
It can be, if the web server user is in the directory's group. But be careful — it means the web server process can write to the directory, which could be exploited if the server is compromised.
