When to use chmod 400

Use 400 for: deployed SSL certificates, SSH authorized_keys files, backup encryption keys, published SSH public keys you don't want to accidentally overwrite, and any credential file that should never be modified in place.

Security note: 400 is extremely safe. It's one of the most restrictive permissions. The trade-off is that you need to chmod it writable to make any changes.

Consider instead: If you need to edit the file: use 600. If others should read too: use 444. If it's an executable: use 500.

Common Mistake

Using 400 on files that are part of an automated process that needs to write to them (like log files or temp files). The process will fail with 'Permission denied' on write operations.

chmod 400 vs 600

400 vs 600
400 vs 600: 400 is read-only for the owner, 600 adds write permission. Use 400 for files you want to protect from accidental modification. Use 600 for files you actively edit (like SSH private keys you might regenerate).

Frequently Asked Questions

When is 400 better than 600?
When you want to prevent accidental writes. A deployed SSL private key should be 400 because there's no reason to modify it in place. If you need to replace it, you'd delete and recreate rather than edit.
Should authorized_keys be 400?
Either 400 or 600 works. 400 prevents you from accidentally adding keys without explicitly changing permissions first. 600 is more convenient if you frequently add/remove keys. SSH accepts both.
Can I still delete a 400 file?
Yes, if you have write permission on the parent directory. File deletion is controlled by directory permissions, not file permissions.
