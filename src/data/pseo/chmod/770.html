When to use chmod 770

Use 770 for: shared project directories, team build directories, deployment staging areas, and any workspace where a defined group needs full access but the rest of the system shouldn't see it.

Security note: 770 is safe when group membership is controlled. Every user in the group gets full access, so make sure group membership is intentional.

Consider instead: If others need read access: use 775. If only the owner needs access: use 700. For files (not directories): use 660.

Common Mistake

Forgetting to add team members to the correct group. Users who aren't in the file's group are treated as 'others' and get zero access under 770.

chmod 770 vs 775

770 vs 775
770 vs 775: 770 blocks others entirely, 775 gives others read+execute. Use 770 for private team resources, 775 when the content should be publicly readable but only team-editable.

Frequently Asked Questions

How do I add users to a group for 770?
Use: sudo usermod -aG groupname username. The user needs to log out and back in for the change to take effect. Verify with: groups username.
Should I use 770 or 700 for project directories?
Use 770 if multiple people work on the project. Use 700 if you're the only one. On a personal laptop, 700 is fine. On a shared server, 770 with a proper group is the way to go.
Does SGID matter with 770?
Yes. Set SGID with chmod g+s on 770 directories so new files inherit the group. Without it, files created by different users get different groups, breaking shared access.
